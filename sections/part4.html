14. Data Schemas & Services

Database Tables

Two PostgreSQL tables. All rows inherit id (UUID PK), created_at, updated_at from the SQLAlchemy base model.

dg_gold_seed_runs — one row per seed
scenario_idStringUnique seed ID, e.g. seed_00001. Indexed.
scenario_content_hashStringSHA hash — used to skip unchanged seeds on sync.
sourceStringlibrary (from disk) or custom (user-entered).
raw_intentTextThe plain-text intent from the seed.
scenario_jsonJSONFull seed object: scenario_id, intent, domain, type.
jinja_candidates_jsonJSONStage 2 output — array of Jinja2 MAL candidates.
generation_jsonJSONStage 3 output — array of LLM-generated MAL records.
validation_jsonJSONStage 4 output — per-record results for all 5 stages.
normalized_record_jsonJSONFinal clean record — ready for training corpus.
dg_llm_expansion_batches — one row per LLM call
scenario_idStringWhich seed this batch belongs to.
statusStringpreview / completed / failed
dry_runBooleanTrue = prompt saved, no LLM call made.
num_variants_requestedIntegerHow many records were requested.
prompt_system / prompt_userTextExact prompts sent to the LLM — full audit trail.
model_nameStringLLM model used, e.g. gpt-4o.
items_jsonJSONParsed LLM output — array of MAL records.
prompt_tokensIntegerInput token count (null for dry runs).
completion_tokensIntegerOutput token count (null for dry runs).
latency_msIntegerWall-clock LLM call time in ms.
error_messageTextError detail if status is failed.

Services Directory

19 service files in backend/app/services/. All are domain-agnostic — domain behavior is configured via data artefacts (seed JSONL, domain pack JSON, templates), not hardcoded.

llm_mal_from_intent.pyStage B — generates MAL from each accepted intent. Retries failures.
llm_dataset_orchestrator.pyOrchestrates Stage A → filter → Stage B flow and batch lifecycle.

llm_prompt_build.pyConstructs system + user prompts from parameters. No domain coupling.

15. Tech Stack — Why Each Tool Was Chosen

TL;DR Every tool was chosen to match or complement the Metafore platform stack. The frontend is a direct subset of Varnam's UI layer. The backend mirrors Metafore's FastAPI + PostgreSQL + Pydantic pattern. One exception: Jinja2, which is actively being researched for replacement.
🔗

Metafore Platform Alignment: The ALM Data Gen platform deliberately mirrors the Metafore/Varnam tech stack — same frontend primitives (React, Vite, Radix UI, Lucide, TanStack Query, Tailwind), same backend pattern (FastAPI, Pydantic, PostgreSQL, Docker). This means any Metafore engineer can read and contribute to this codebase without a new tool learning curve.
