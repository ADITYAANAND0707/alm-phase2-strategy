14. Data Schemas & Services

Database Tables

Two PostgreSQL tables track pipeline state. All tables inherit id (UUID PK), created_at, and updated_at from the SQLAlchemy base.

dg_gold_seed_runs — Seed lifecycle (one row per seed)

ColumnTypeDescription
scenario_idString (unique)Unique seed identifier, e.g. seed_00001
scenario_content_hashStringSHA hash of seed content — used to detect changes and skip unchanged seeds on sync
sourceStringlibrary (from disk) or custom (user-entered)
raw_intentTextThe intent string from the seed
scenario_jsonJSONFull seed object (scenario_id, intent, domain, type)
jinja_candidates_jsonJSONStage 2 output — array of Jinja2 MAL candidates
generation_jsonJSONStage 3 output — array of LLM-generated MAL records
validation_jsonJSONStage 4 output — per-record validation results (all 5 stage results)
normalized_record_jsonJSONFinal clean record — ready for training corpus

dg_llm_expansion_batches — LLM call tracking (one row per batch)

ColumnTypeDescription
scenario_idStringWhich seed this batch belongs to

statusStringpreview (saved prompt, not executed) / completed / failed
dry_runBooleanTrue = prompt saved but no LLM call made
num_variants_requestedIntegerHow many records were requested
seed_jsonJSONSeed object at time of generation
jinja_candidates_jsonJSONJinja2 candidates used as LLM examples
prompt_systemTextExact system prompt sent to LLM
prompt_userTextExact user prompt sent to LLM
model_nameStringLLM model used, e.g. gpt-4o
raw_response_textTextRaw LLM response text before parsing
items_jsonJSONParsed LLM output — array of MAL records
error_messageTextError detail if status is failed
prompt_tokensIntegerInput token count (null for dry runs)
completion_tokensIntegerOutput token count (null for dry runs)
latency_msIntegerWall-clock LLM call time in milliseconds

Services Directory

All service files are in backend/app/services/. All are domain-agnostic in code — domain is configured via data artefacts, not hardcoded.

FileGroupWhat it does
jinja_expand.py Stage 2 Renders Jinja2 templates with domain vocabulary. Pure template rendering — no MAL coupling.
llm_jinja_generate.py Stage 3 — Mode A Jinja2-First orchestrator. Builds prompt from Jinja2 candidates, runs LLM generation, parses output.
llm_expand_runner.py Stage 3 — Mode A Batch execution runner. Manages preview → execute lifecycle, handles stored batch execution.
llm_intent_synthesize.py Stage 3 — Mode B Intent-First Stage A. Generates intent variants from seed anchors + Jinja2 context. Domain-filters and deduplicates.
llm_mal_from_intent.py Stage 3 — Mode B Intent-First Stage B. Generates valid MAL from each accepted intent. Retries failures before discard.
llm_dataset_orchestrator.py Stage 3 — Mode B Intent-First orchestration. Coordinates Stage A → filter → Stage B flow. Manages overall batch lifecycle.
llm_prompt_build.py Stage 3 — Shared Constructs system + user prompts from inputs. No MAL or domain coupling — prompt is assembled from parameters.
llm_response_parse.py Stage 3 — Shared Parses structured LLM output into typed records. Generic — no domain awareness needed at parse time.
llm_openai_compatible.py Stage 3 — Shared HTTP client for OpenAI-compatible APIs. Handles auth, retries, token accounting. Domain-agnostic.
llm_catalog_generate.py Stage 3 — Catalog Catalog-level generation — runs LLM generation from database records (not from Jinja2 candidates).
llm_catalog_records.py Stage 3 — Catalog Catalog record handling and formatting for catalog-mode generation.
llm_training_records.py Stage 3 — Catalog Shapes and normalizes generated output into training-ready record format.
mal_validation_service.py Stage 4 5-stage in-process PEG-parser MAL validator. Runs syntax → schema → policy → dependency → compile_check. Returns per-stage results with error detail.
mal_repair_service.py Stage 4 Auto-repair service. Detects and fixes common MAL structural errors before re-running validation. Reduces reject rate.
diversity_metrics.py Output Normalised Levenshtein Distance computation across corpus. In-memory TTL cache (120s) prevents timeout. ?refresh=true forces recompute.
seed_library_loader.py Stage 1 Loads seeds from data/seed_scenarios/telecom_v1.jsonl. Falls back to pipeline/seeds/seeds.json if disk path not found.
seed_pipeline_sync.py Stage 1 Upserts disk seeds into dg_gold_seed_runs. Uses content hash for change detection — skips unchanged seeds.
seed_content_hash.py Stage 1 Generates SHA hash of seed content for deduplication and change detection.
pipeline_bridge.py Utility Bridge utility between pipeline stages. Small helper module for stage-to-stage data handoff.

15. Tech Stack — Why Each Tool Was Chosen

TL;DR Every tool was chosen to match or complement the Metafore platform stack. The frontend is a direct subset of Varnam's UI layer. The backend mirrors Metafore's FastAPI + PostgreSQL + Pydantic pattern. One exception: Jinja2, which is actively being researched for replacement.
🔗

Metafore Platform Alignment: The ALM Data Gen platform deliberately mirrors the Metafore/Varnam tech stack — same frontend primitives (React, Vite, Radix UI, Lucide, TanStack Query, Tailwind), same backend pattern (FastAPI, Pydantic, PostgreSQL, Docker). This means any Metafore engineer can read and contribute to this codebase without a new tool learning curve.
