13. Governance & Engineering Standards

Dataset Governance

Prevent Silent Drift

Version the dataset, registry, templates, prompts, and model choices separately. Any change to a template or prompt creates a new version of the associated records.

Decision Log

Remember the Why

Track confirmed choices, deferred choices, and rationale in the Decision Log (Section 19). Future team members need to understand why approaches were chosen, not just what they are.

Acceptance Criteria

No Subjective Approvals

Use the checklist in Execution Playbook (Section 10.5). Every record that enters the registry must pass the same written criteria — update the checklist as new quality issues emerge.

Provenance Tracking

Full Traceability

Every accepted record stores its origin: which seed, which template, which prompt version, and which reviewer approved it. This metadata enables targeted correction when quality issues surface.

Failure Analysis

Learn From Rejects

Keep a reject log with reason codes and sample examples. Failure patterns improve later stages faster than only counting accepted records — they reveal template or prompt weaknesses directly.

Promotion Workflow

Separate Draft from Trusted

Lifecycle states: draft → validated → reviewed → accepted → archived. Only accepted records are used for fine-tuning and evaluation.

14.4 Registry Lifecycle

StatusMeaningWhen It Moves Forward
draftFreshly generated from seed, Jinja2, or LLMAfter automated validation starts
validatedPassed parser, policy, and compile checksAfter novelty and duplicate review
reviewedSeen by a reviewer and scoredAfter acceptance decision
acceptedApproved for registry useAvailable for prompting, evaluation, and later adaptation
archivedKept for history but not active for current usageWhen records are replaced or superseded

13b. API Quick Reference

One-line reference for every live backend endpoint. Full interactive docs at http://localhost:8010/docs (Swagger UI).

MethodEndpointPurpose
GET/healthBackend health check — returns 200 when the service is running
GET/docsFastAPI Swagger UI — interactive API explorer
POST/api/v1/seeds/sync-librarySync seed scenarios from disk (data/seed_scenarios/*.jsonl) into the database
POST/api/v1/seeds/ingest-customAdd a one-off custom scenario directly (bypasses disk files)
POST/api/v1/jinja2/expandRun Jinja2 expansion for a specific scenario — returns 6 task-type variants
POST/api/v1/llm/expandBuild prompt and optionally call LLM for one seed scenario
POST/api/v1/llm/execute-latest-previewExecute the most recently saved dry-run (Prompt Saved) batch
POST/api/v1/llm/batches/{id}/executeExecute a specific stored preview batch by ID
GET/api/v1/llm/expansionsList all LLM expansion batches with status, token counts, and latency
POST/api/v1/llm/generate-from-databaseFull catalog generation — runs LLM across all stored seeds in one batch
POST/api/v1/validateRun the 5-stage MAL quality gate (syntax → schema → policy → dependency → compile_check)
GET/api/v1/registryList all validated training records — supports filter by task type and domain

All endpoints return JSON. Error responses follow FastAPI’s standard 422/500 format. Token and latency fields appear on LLM batch responses.
