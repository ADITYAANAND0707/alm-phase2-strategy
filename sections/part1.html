1. Executive Summary

The ALM platform generates, validates, and manages enterprise application models written in MAL — a purpose-built domain language. It already handles user intent, live validation, and versioned persistence. What it lacks is an offline data factory: a repeatable pipeline that creates high-quality training and evaluation examples at scale.

Phase 2 builds that factory in five stages. Stage 1: a curated library of 56 gold seed examples. Stage 2: a Jinja2 template engine that produces 336 balanced variants from those seeds — the safest, most explainable expansion method. Stage 3: GPT-4o synthesizes rich, diverse MAL records from each seed. Stage 4: a 5-step quality gate validates every record (syntax → schema → policy → dependency → compile). Stage 5: validated records land in the training registry, ready for fine-tuning and evaluation.

  • Scope: MAL only. Phase 2 generates examples for application, entity, workflow, policy, agent, integration, event, and evolution. UI constructs are excluded from the first data wave.
  • All 5 stages live. The full pipeline is built and demo-ready — 56 canonical gold seeds, 336 Jinja2 variants, GPT-4o AI Synthesis with token/latency tracking, 5-stage Quality Gate, and a browsable Training Registry with export. Full React/Vite/TypeScript frontend dashboard at http://localhost:5173.
  • PR #3 merged. All pipeline work is in the shared team repo (DIVYAMETAFORE25/ALM-Data-Gen). The strategy doc is kept separately on Aditya’s branch.
  • This document is a living report. Update the progress tracker, decision log, coverage grid, and example library as Phase 2 matures toward fine-tuning experiments.

2b. Demo Playbook

A rehearsed demo is the fastest way to communicate the value of this project. Two scripts — one for managers (5 minutes, no code), one for a technical deep-dive (2 minutes, code visible).

Manager Demo — 5 Minutes, No Code

Step 1 — Open Dashboard
  • Navigate to http://localhost:5173
  • Point to the Pipeline Story Card: animated counts show 56 seeds → 336 Jinja2 variants → N AI records → M validated
  • Say: "Each number represents a stage of an automated data factory we built for training AI models on ALM."
Step 2 — Show AI Synthesis
  • Click AI Synthesis in the sidebar
  • Expand any batch row → click the Records tab
  • Show seed intent (plain English) vs. generated MAL (structured code) side-by-side
  • Say: "GPT-4o reads a simple intent and generates a full MAL application description. We track every token used and every millisecond of latency."
Step 3 — Validate a Record
  • Click the Validate → button on any AI-generated record
  • On the Quality Gate page, click Run Validation Pipeline
  • Show 5 stages passing: syntax ✓, schema ✓, policy ✓, dependency ✓, compile ✓
  • Say: "Before any record enters the training registry, it passes 5 automated quality checks. Nothing gets in unless it's correct."
Step 4 — Export Demo Report
  • Click Training Registry → click Demo Report button
  • Hand over the downloaded Markdown summary
  • Say: "This is the full pipeline run summary — ready to share with stakeholders or include in a project report."

Technical Demo — 2 Minutes, Code Visible

Step 1 — Seed to Variant
  • Open Deterministic Expansion → click Run Jinja2
  • Expand a row → show the MAL code, task type badge (A–F), and record_id format
  • Point to the 6 task-type pills (Create / Modify / Repair / Explain / Policy / Disambiguate)
Step 2 — LLM Batch Execution
  • Open AI Synthesis → click Execute Latest Saved Prompt
  • Show the token badge updating: prompt tokens / completion tokens / latency ms
  • Open the Prompt tab on any batch to show system prompt + user prompt structure
Step 3 — Quality Gate Pipeline
  • Open Quality Gate → load MAL from AI Synthesis picker
  • Run pipeline → show stage-by-stage results with structured error messages and line numbers
  • Optionally show a failing case to demonstrate error reporting depth

4. Scope Definition

Phase 2 synthetic-data work is MAL-specific and tightly bounded. What is in scope, what is excluded, and what the objective is — so the team stays aligned as the pipeline grows.

All 5 stages liveSeed Library (56 seeds) → Deterministic Expansion (336 Jinja2 variants) → AI Synthesis (GPT-4o) → Quality Gate (5-stage validation) → Training Registry (filterable, exportable).
Next stepPromote-to-Registry endpoint, human annotation UI, and fine-tuning dataset export in instruction-following JSONL format.
In Scope — Phase 2
  • MAL only — the primary language for this data wave
  • application — top-level MAL construct
  • entity — data model definitions with attributes
  • workflow — step sequences and trigger logic
  • policy — business rules and constraint enforcement
  • agent — autonomous task orchestration
  • integration — external system connections
  • event — event-driven patterns
  • evolution — schema migration and change records
  • Objective: validator-approved intent→MAL pairs for fine-tuning & evaluation
Out of Scope — Phase 2
  • UI constructs — screens, components, actions, navigation not in first data wave
  • Full-parameter retraining — efficient fine-tuning is the first adaptation step
  • Multi-domain data — banking, healthcare seeds deferred to Phase 3
  • Active learning loop — model-gap-driven generation is a future roadmap item
  • LLM-as-judge validation — semantic quality scoring planned but not yet built
  • Annotation platform — human review UI is a next-build item, not current sprint
