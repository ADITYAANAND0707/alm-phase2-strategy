1. Executive Summary

🏭
The ALM Data Generation platform is a 4-stage synthetic training data factory for MAL — the formal domain language used by the ALM platform. All four stages are now fully live and integrated. The system takes 50 hand-authored telecom seed scenarios, produces deterministic Jinja2 template expansions, synthesizes diverse AI-generated records via GPT-4o, and gates everything through a 5-stage in-process MAL validator before storage. The full pipeline is accessible through a React/Vite frontend with 8 routes and a FastAPI backend with 23 mounted endpoints.
What was built
4-Stage Pipeline — End to End
  • Stage 1: 50-seed telecom library with sync API and per-seed DB lifecycle tracking
  • Stage 2: Jinja2 deterministic expansion — 4 templates, 1 domain pack, reproducible output
  • Stage 3: LLM synthesis in 2 modes — Jinja2-First and Intent-First (2-pass)
  • Stage 4: 5-stage PEG-parser MAL validation + auto-repair service
  • Training registry, record browser, and diversity metrics (NLD + TTL cache)
Additional engineering delivered
Backend Reliability & Coverage
  • 7 integration test modules — first test coverage in the project
  • In-memory TTL cache (120s) on diversity metrics to prevent NLD timeouts
  • MAL repair service — auto-fixes common structural errors before re-validation
  • Token + latency stored per LLM batch in dg_llm_expansion_batches
  • Fixed sidebar navigation (fixed position, collapsible secondary items)
  • React key warning fix, <a> → <Link> SPA navigation fixes
  • Team handoff documentation + research context pack in docs/research/
🔬

Where we are now: The prototype is working end-to-end. Research is underway (ticket RE-474) to determine the next major architectural step — replacing Jinja2 Stage 2 with a smarter generation layer, and making the LLM expansion stage production-grade with quality-gated loops and multi-signal scoring.

Design Principle (non-negotiable)

💡

Think of it as: Grammar = rules, Domain Pack = vocabulary, Guidelines = style guide. Swap all three data files to switch domains — zero engine code changes needed.

Domain Plugin Architecture
📐
Grammar
Hard structural constraint — only valid MAL structures pass the quality gate. Embedded in mal_validation_service.py as an in-process PEG parser.
📦
Domain Pack
Vocabulary config — entity names, field types, enum values, relations. Current: pipeline/domain_packs/telecom.json. Swapping this file changes the domain.
📋
Guidelines
Generation hints — preferred patterns, anti-patterns, domain rules. Currently implicit in Jinja2 templates and LLM prompts. Target: explicit plugin artefact.
⚙️
Engine code
All service and route code is domain-agnostic. Verified: only 4 data artefacts are domain-specific — seed JSONL, domain pack JSON, template files, grammar. Zero engine code changes needed to add a new domain.

LLM Modes
Jinja2-First + Intent-First
Mode A (Jinja2-First): Jinja2 candidates used as structured examples for the LLM. Mode B (Intent-First): 2-pass — Stage A generates intent variants from seeds, Stage B generates MAL from each accepted intent. Every saved record is a fresh (intent, MAL) pair — seed intent never reused as output.

Stage-by-Stage Status

Stage 1
Seed Library
50 telecom seeds — Live
50 hand-authored real-world telecom scenarios — the source of truth for everything downstream.
Hand-authored seed scenarios are the grounded starting point. Each seed defines intent and task type. Stored in telecom_v1.jsonl on disk, synced to PostgreSQL, full lifecycle tracked per seed. Frontend: /seeds page with expand, validate, repair, bulk generate controls.
telecom_v1.jsonl seed_library_loader.py seed_pipeline_sync.py dg_gold_seed_runs
Stage 2
Deterministic Expansion
Jinja2 templates + domain pack — Live
Templates automatically turn 1 seed into 5–10 structural MAL variants — same input, same output, every time.
Each seed is expanded into 5–10 MAL candidates using 4 Jinja2 template files filled with telecom vocabulary. Running the same seed twice always produces identical output — deterministic, debuggable, reproducible. Candidates feed Stage 3 as structured examples. CLI: python pipeline/generate.py.
jinja_expand.py entity.j2 / workflow.j2 policy.j2 / evolution.j2 telecom.json
Stage 3
AI Synthesis
GPT-4o, 2 modes — Live
GPT-4o reads the seed variants and writes novel, semantically rich MAL records — in two distinct modes.
Two generation modes. Mode A (Jinja2-First): Jinja2 candidates as LLM examples → diverse MAL variants. Mode B (Intent-First): LLM generates fresh intents from seed anchors (Stage A), then MAL from each intent (Stage B). Every batch stored with full prompt, output, tokens, latency. Dry run previews prompts for free.
llm_jinja_generate.py llm_intent_synthesize.py llm_mal_from_intent.py llm_dataset_orchestrator.py
Stage 4
Quality Gate
5-stage PEG validation + repair — Live
Every record is parsed against the MAL grammar — rejected immediately if any rule breaks. Auto-repair catches common errors first.
Every generated MAL record passes 5 sequential checks in-process (no external dependency). Fail at any stage → rejected with exact rule and line. Auto-repair service catches and fixes common structural errors before re-validation. Bulk validation endpoint processes entire batches. Frontend: /validation with Batch Browser + Single Record + All Records tabs.
mal_validation_service.py mal_repair_service.py POST /validate/bulk POST /validation
