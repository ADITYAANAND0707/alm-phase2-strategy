6. Task Coverage & Construct Map

Every seed scenario belongs to one of six task types. Each task type represents a distinct way a developer uses MAL — creating a new entity, modifying an existing one, repairing broken logic, etc. The pipeline generates training records across all six types to ensure balanced corpus coverage.

Task Type
Create
Define a new MAL entity, workflow, or policy from scratch

Example: "Build a subscriber onboarding system with billing integration." The LLM must generate a complete valid MAL application definition.

Task Type
Modify
Extend or update an existing MAL structure

Example: "Add a loyalty points field to the existing Subscriber entity." Requires understanding current structure before evolving it.

Task Type
Repair
Fix a broken or invalid MAL definition

Example: "This MAL entity has an invalid relationship reference — fix it." Tests the model's ability to diagnose and correct grammar violations.

Task Type
Explain
Describe what a given MAL definition does in plain English

Example: "What does this MAL workflow describe?" Tests comprehension, not generation — useful for instruction-tuned models.

Task Type
Policy
Define or reason about MAL access control rules

Example: "Add a policy that restricts PlanType changes to admin roles only." Requires understanding MAL's policy block structure and conditions.

Task Type
Disambiguate
Resolve ambiguous intent into a specific MAL choice

Example: "The user said 'track plan changes' — should this be a workflow trigger or an audit policy?" Tests reasoning under ambiguity.

Construct Coverage Heatmap

MAL has four primary construct families. Each Jinja2 template covers one family. Coverage tracks which task types have templates for which constructs.

Construct Create Modify Repair Explain Policy Disambiguate Template file
Entity Full Full Full Full Part Part entity.j2
Workflow Full Full Part Full — Part workflow.j2
Policy Full Part Part Full Full Part policy.j2
Evolution — Full Full Part — — evolution.j2

Full = complete template coverage  |  Part = partial coverage  |  — = no template yet

Stage 1 — Seed Library

The seed library is the grounded starting point for all generated data. Every output record traces back to a human-authored seed scenario. Seeds define intent, task type, and domain — these three fields drive everything downstream.

Seed schema

JSONL — telecom_v1.jsonl (one line = one seed)
{
  "scenario_id": "seed_00001",
  "intent":      "Build a subscriber onboarding system",
  "domain":      "telecom",
  "type":        "Create"
}

Key facts

  • 50 seeds in data/seed_scenarios/telecom_v1.jsonl
  • Fallback: pipeline/seeds/seeds.json (6 task types, used if disk path not found)
  • Sync API: POST /api/v1/seeds/sync-library — upserts disk seeds into PostgreSQL
  • Ingest custom: POST /api/v1/seeds/ingest-custom — adds user-defined seeds
  • DB table: dg_gold_seed_runs — one row per seed, full pipeline lifecycle
  • Frontend: /seeds page — expand, validate, repair, bulk generate controls

Stage 2 — Deterministic Jinja2 Expansion

Each seed is matched to one or more Jinja2 templates and filled with vocabulary from the domain pack. The output is 5–10 MAL candidate strings. Same seed + same domain pack = same output every time. No LLM involved.

How it works

  • Template files: entity.j2, workflow.j2, policy.j2, evolution.j2
  • Domain vocabulary: pipeline/domain_packs/telecom.json — entity names, field types, enum values, workflow verbs
  • Rendering service: backend/app/services/jinja_expand.py — domain-agnostic, pure template rendering
  • CLI: python pipeline/generate.py — writes to data/jinja2_expansion/
  • API: POST /api/v1/jinja2/expand

Why deterministic matters

  • Bugs are reproducible — same input always gives same output
  • Regression testing is trivial — compare before/after output files
  • No token cost — template rendering is near-instant and free
  • Provides structured context for the LLM in Stage 3 — better than starting from nothing
  • Domain change = swap one JSON file, run again

Stage 3 — AI Synthesis (Two Modes)

Stage 3 is where creative diversity is added. The LLM takes structured inputs and generates new MAL records that follow the grammar but explore a broader space than templates alone. Two modes are available.

Mode A — Jinja2-First
Standard generation
  • Jinja2 candidates used as structured examples for the LLM
  • LLM generates new MAL variants that stay close to template patterns
  • Good for: conservative expansion, high pass rate, faster runs
  • Orchestrator: llm_jinja_generate.py
  • Route: POST /api/v1/llm/generate-from-jinja
Mode B — Intent-First
Primary AI mode
  • 2-pass: Stage A generates intents, Stage B generates MAL from each intent
  • Every saved record is a fresh (intent, MAL) pair — seed intent never reused
  • Domain-filtered and deduplicated between Stage A and B
  • Failed records retried automatically before discard
  • Orchestrator: llm_dataset_orchestrator.py

Shared LLM controls (both modes)

Prompt management
  • Dry run toggle — preview prompt, no LLM call, zero cost
  • Save prompt for review — stores as preview batch
  • Custom system instructions override
  • Task type filter (select which types to generate)
Generation controls
  • Temperature slider
  • Chunk size (Jinja2 candidates per call)
  • Max tokens per completion
  • Context sample size
Observability (stored per batch)
  • prompt_system + prompt_user — exact text sent
  • prompt_tokens + completion_tokens
  • latency_ms — wall-clock LLM call time
  • status: preview / completed / failed

Stage 4 — Quality Gate

Every generated record must pass all 5 validation stages before it enters the training corpus. Validation runs in-process — no external service dependency. Fail at any stage and the record is rejected with the exact rule and line that failed.

5-stage validation pipeline

1
Syntax — Is the MAL parseable by the embedded PEG grammar?
2
Schema — Do all entity fields have valid types?
3
Policy — Are policy rules internally consistent?
4
Dependency — Are all referenced entities defined?
5
Compile check — Would this compile in the ALM runtime?

Services and routes

  • mal_validation_service.py — in-process PEG validator (5 stages)
  • mal_repair_service.py — auto-fixes common MAL structural errors
  • POST /api/v1/validate/bulk — validate many records, optional DB write-back
  • POST /api/v1/validation — validate a single record, self-contained

Frontend: /validation (Master Validator)

  • Batch Browser tab: select LLM batch or Jinja2 session → validate all records
  • Single Record tab: load from seeds/Jinja2/LLM/paste → per-stage result tabs
  • All Records tab: bulk validate all stored Jinja2 or LLM output

8. Stage 2 Rationale & Transition

💡
Jinja2 was the right first choice. It gave the pipeline deterministic structure, low cost, easy debugging, and a reproducible baseline — all without LLM involvement. But as the system grows toward multiple domains and production-grade generation, Jinja2's fixed-template model becomes the bottleneck.
Why Jinja2 was right at Stage 1
  • Zero LLM cost — template rendering is instant and free
  • Deterministic output — same input always gives same result
  • Easy to debug — template fill is transparent, no black box
  • Provides structured context for Stage 3 — better LLM output than zero-shot
  • Proved the pipeline worked before spending tokens
  • Domain change = swap one JSON vocabulary file
Why Jinja2 becomes a bottleneck
  • Coverage is fixed — new MAL structures require new template files
  • All variants from the same template look structurally similar — low structural diversity
  • Adding a new domain means authoring 3–4 template files from scratch
  • Cannot generate novel structural combinations — fills slots, does not invent
  • LLM in Stage 3 ends up "correcting" template patterns instead of exploring
  • Guidelines (which patterns to prefer) are baked implicitly into templates — not explicit
🔬

RE-476 research goal: Find a replacement for Stage 2 that is domain-agnostic in engine code but domain-aware in behavior — using a Grammar + Domain Pack + Guidelines plugin contract. When a domain is plugged in, the system generates output that is structurally correct, semantically meaningful, and contextually smart. See Part III for the full research framing.
