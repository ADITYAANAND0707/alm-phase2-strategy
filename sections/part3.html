11. Stage 2 Upgrade — Moving Beyond Templates RE-476 — In Research

⚠️

The problem in one sentence: Jinja2 templates are rigid — adding a new MAL construct or domain means manually writing a new template file, which doesn't scale and bakes domain logic into code.

🎯
Goal: Replace fixed Jinja2 templates with a generation engine that is domain-agnostic in code (engine never changes per domain) but domain-aware in behavior (when Grammar + Domain Pack + Guidelines are plugged in, output is structurally correct, semantically meaningful, and contextually smart for that domain).

The Plugin Contract — What Any Solution Must Support

🔗

Already defined in Part I — Design Principle. The plugin contract (Grammar + Domain Pack + Guidelines) is the non-negotiable scope constraint for this research. Any candidate approach must satisfy all three artefacts — engine code never changes per domain, only the data files do.

⚙️

Research constraint: Every candidate approach in this section is evaluated against the same question — can Grammar, Domain Pack, and Guidelines each be expressed as a loadable data artefact that the engine consumes without code changes?

Candidate Approaches

Approach 1
Grammar-Based Generation (PCFG / Earley Sampler)

Use a probabilistic context-free grammar (PCFG) or an Earley parser to generate random valid MAL strings directly from the grammar definition. Domain vocabulary is injected into terminal productions. No LLM cost at Stage 2.

Strengths
  • Fully deterministic or controlled-random
  • Zero LLM cost
  • Grammar enforcement is exact — only valid structures produced
  • Coverage limited only by grammar breadth
Weaknesses
  • Requires authoring a formal grammar (EBNF/PCFG) for each target language
  • Output can be syntactically valid but semantically nonsensical without good domain vocabulary injection
  • Guidelines are hard to encode into grammar rules

Plugin contract: Grammar = EBNF/PCFG file. Domain Pack = terminal symbol vocabulary. Guidelines = production weights (probabilistic grammar).

Reference: NLTK's generate module, Lark parser, Python pegen. Used in fuzzing tools (AFL grammar mode) and structured test data generation.

Approach 2
Constrained LLM Generation — Outlines / DSPY

Use a constrained decoding library to force an LLM to generate output that conforms to a grammar or schema at the token level. The LLM provides semantic richness; the constraint enforces structural validity.

Outlines (dottxt-ai/outlines)
  • Grammar-constrained decoding via EBNF — generated tokens must form valid grammar
  • Supports Regex, JSON Schema, EBNF grammar constraints
  • Works with local models (llama.cpp, transformers) — no API cost
  • Grammar acts as a hard filter on token probabilities
DSPY (Stanford)
  • Declarative LLM program — define what output looks like, not how to prompt
  • Typed signatures enforce output structure
  • Optimizable — can tune prompts automatically using examples
  • Works with any LLM API (GPT-4o compatible)

Plugin contract: Grammar = EBNF grammar file (Outlines) or typed signature (DSPY). Domain Pack = few-shot examples or context injection. Guidelines = DSPY module constraints or Outlines regex guards.

Key question for research: Can Outlines enforce the MAL PEG grammar at generation time? Can DSPY be domain-agnostic while domain-aware through typed signatures loaded from config?

Approach 3
Hybrid — Grammar Sampler + LLM Semantic Fill

A two-step approach: (1) a grammar sampler generates the structural skeleton of a MAL record, (2) an LLM fills semantic content into the skeleton using domain vocabulary and guidelines. Combines grammar-exact structure with LLM-quality semantics.

Plugin contract: Grammar = structure-skeleton generator. Domain Pack = LLM vocabulary context. Guidelines = LLM prompt instructions for semantic fill.

Advantage: Best of both worlds — guaranteed structural validity + semantic richness. Cost: Requires LLM call per record at Stage 2 (unlike pure grammar sampling or Jinja2).

12. Making AI Synthesis Smarter — Research Agenda RE-477 — In Research

⚠️

The problem in one sentence: The current LLM stage runs once, uses a static prompt, and scores output with a single signal (string edit distance) — it cannot learn from its own quality results or adjust between runs.

🏗️
Goal: Evolve Stage 3 from a useful prototype (single-pass, static prompt, NLD-only quality) to a production-grade data generation service (quality-gated loops, multi-signal scoring, cost visibility, operator observability). "Production-grade" here does not mean infinite scale — it means adaptive, observable, and reliably improving.

What Platform-Grade Requires

Quality Assurance
Multi-Signal Scoring

Output scored on 3+ signals before entering corpus. NLD diversity + semantic validity (classifier or embedding distance) + grammar coverage (are edge cases represented?) + utility proxy (perplexity under reference model).

Adaptivity
Quality-Gated Loop

If a generated chunk scores below quality threshold, the loop adjusts (temperature, prompt framing, chunk size) and regenerates before accepting or discarding. System improves across runs without manual tuning.

Observability
Cost + Health Visibility

Per-run: tokens used, cost ($), pass rate, diversity score — surfaced in the UI. Alert if pass rate drops below threshold or cost per record spikes. Data already exists in dg_llm_expansion_batches — only the UI surface is missing.

✅

Lowest hanging fruit (no architecture change needed): Wire the 3 missing frontend routes + surface token/cost data already in the database. Together these close 2 gaps in under a day of frontend work.
