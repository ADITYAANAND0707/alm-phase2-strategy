22. Glossary

MAL (rhymes with "pal")
Meta Application Language — a formal language for describing software applications as entities, workflows, and policies. The target language of this training data factory.
Seed Scenario
A short human-authored record (intent + task type + domain) that is the grounded starting point for all generated data. Every output traces back to a seed.
Jinja2
A Python templating engine. Used in Stage 2 to fill MAL structure templates with domain vocabulary — like mail-merge for code. Deterministic: same input = same output.
Domain Pack
A JSON vocabulary file for a specific industry domain. The telecom pack (telecom.json) contains entity names (Subscriber, PlanType), workflow verbs, enum values, and policy conditions. Swapping it changes the domain.
Intent-First Mode
Stage 3 Mode B — a 2-pass LLM generation strategy where Stage A generates intent variants from seed anchors and Stage B generates MAL from each accepted intent. Every saved record is a fresh (intent, MAL) pair; seed intent is never reused as output.
Jinja2-First Mode
Stage 3 Mode A — the standard LLM generation strategy where Jinja2 template candidates are used as structured examples in the LLM prompt. LLM generates new MAL variants that stay close to template patterns.
PEG Parser ("peg")
Parsing Expression Grammar parser — the embedded MAL grammar validator in mal_validation_service.py. Runs in-process with no external service dependency. Checks 5 sequential stages: syntax → schema → policy → dependency → compile_check.
NLD (say each letter: N-L-D)
Normalised Levenshtein Distance — measures string similarity between 0 (identical) and 1 (completely different). Used by the diversity metrics endpoint to score how varied the generated corpus is. High NLD = more useful training data.
Dry Run
Preview mode for LLM generation. Builds the full system + user prompt and saves it as a preview batch record — no LLM API call is made. Free and instant. Lets you inspect and adjust prompts before executing.
Plugin Contract
The set of data artefacts an author must write to add a new domain to the pipeline: (1) Seed scenarios JSONL, (2) Domain pack JSON, (3) Generation guidelines. Engine code requires zero changes. Currently implicit; RE-476 research is formalizing it.
Batch
One LLM generation request for one seed scenario. Stored in dg_llm_expansion_batches with full prompt, output, token usage, and latency. Status: preview → completed / failed.
TTL Cache
Time-to-live in-memory cache. Used on the diversity metrics endpoint (120s TTL) to prevent NLD computation timeout on large corpora. Use ?refresh=true to force recompute.
TanStack Query
A React library (formerly React Query) that fetches, caches, and automatically refetches API data. Used throughout the frontend for all backend API calls.
Outlines
An open-source library (dottxt-ai/outlines) for grammar-constrained LLM generation. Enforces an EBNF grammar at the token level — generated output can only form valid grammar strings. RE-476 research candidate for replacing Jinja2 Stage 2.
DSPy ("dee-es-pie")
A Stanford framework for declarative LLM programming. Define typed signatures (what output looks like) instead of raw prompts. DSPy optimizes prompts automatically using examples. RE-476 research candidate.
Fine-tuning
Training a pre-trained LLM on custom data to make it better at a specific task. The validated MAL training records produced by this pipeline are the corpus intended for fine-tuning an ALM-specific model.
