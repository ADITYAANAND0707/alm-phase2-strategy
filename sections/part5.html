16. Risk Register

High
LLM stage still not fully defined — Model choice, prompt strategy, and acceptance logic are all open. Keep this stage explicitly marked as proposed until each is resolved.
High
Invalid MAL risk — More volume creates more invalid outputs unless syntax and validation gates are automated before scaling generation.
Medium
Low-diversity dataset risk — Template expansion can become repetitive if payload generation is too narrow. Diversity criteria must be tracked, not assumed.
Medium
Review bottleneck — Human approval will slow the pipeline if scoring, batching, and rejection reasons are not standardized before scaling review.
Medium
Domain drift risk — UI or unrelated constructs can leak into the dataset if generation is not kept inside the MAL-only grammar scope.
Medium
Graph compile failure — Syntactically valid text can still fail downstream compile expectations. Graph checks must remain part of the acceptance funnel.
Planned
Registry discipline — Accepted records must move through explicit states (draft, validated, reviewed, accepted, archived) rather than accumulating in a flat file.

17. Recommendation

Go
Build the Jinja2 and Python expansion layer first. The most concrete, lowest-risk, and easiest-to-govern Phase 2 starting point. It gives the team a controlled synthetic-data baseline before introducing model-driven variation.
Keep
Reuse existing ALM validation and governance architecture. The repo already models validation, graph reasoning, policy checks, and versioning. Those ideas should drive the synthetic-data pipeline design, not be duplicated separately.
Next
Build the Promote-to-Registry workflow and measure acceptance rate. The pipeline now produces validated records from both Jinja2 and AI Synthesis. The next priority is a formal promotion endpoint, a human annotation UI, and tracking acceptance rate and construct coverage per sprint.

22. Glossary

One-line definitions for every key term used in this document and the codebase. (Appears before Changelog for ease of reference.)

Pipeline Terms & Stages
MAL
Model Application Language — the domain-specific language for describing ALM applications
Every training record output is a MAL document. The authoritative grammar defines the parser-compatible structure.
Core DSL
Gold Seed
Hand-crafted canonical scenario used as ground-truth anchor for the entire pipeline
56 seeds in data/seed_scenarios/telecom_v1.jsonl. Each seed defines: record_id, type, domain, intent, MAL output.
Stage 1
Jinja2 / Deterministic Expansion
Template-based expansion of one seed into 6 task-type variants using domain vocabulary
56 seeds × 6 task types = 336 balanced, grammar-correct MAL candidates. Stage 2.
Stage 2
AI Synthesis / LLM Synthesis
GPT-4o generates rich, diverse MAL training records from seed scenarios
Full prompt tracking, token + latency metrics per batch. Runs from the AI Synthesis page at /llm. Stage 3.
Stage 3
Quality Gate
5-stage MAL validation: syntax → schema → policy → dependency → compile_check
Each stage runs independently and reports structured errors with line numbers. All 5 must pass before promotion. Stage 4.
Stage 4
Training Registry
Final curated store of validated MAL records, ready for fine-tuning or evaluation
Filterable by task type and domain. Exportable as JSON. Demo Report download available. Stage 5 at /registry.
Stage 5
Operational & Technical Terms
Task Type
One of six application operations every record is tagged with
Create (A) · Modify (B) · Repair (C) · Explain (D) · Policy (E) · Disambiguate (F)
Domain Pack
JSON vocabulary file for a domain (e.g., telecom.json)
Contains entity names, roles, enums, triggers, and relationships. Fills vocabulary slots in Jinja2 templates.
Batch
One LLM API call result; one batch per seed scenario
Stored with full prompt, token counts, latency, and individual records accessible via Records tab.
Prompt Saved / Dry Run
Batch mode where prompt is built & stored but LLM not yet called
Used for prompt review before spending tokens. Execute separately when ready.
Construct
A structural element of MAL
application · entity · workflow · policy · agent · integration · event · evolution
record_id
Unique identifier for a training record encoding full provenance
Format: seed_NNNNN_jNN_construct_entity_slug. Encodes seed, variant number, construct, and entity without expanding payload.
Token
Unit of LLM input/output text. Prompt tokens + completion tokens per batch.
Used to estimate cost and context usage. Displayed as badge on each batch row in AI Synthesis.
Demo Report
Downloadable Markdown summarizing the full pipeline run
Seed counts, variant counts, LLM records, validation results. Generated from Training Registry page.
TrainingRecord
Canonical data structure for one pipeline output
Fields: record_id, source, stage, domain, task_type, input intent, output MAL document.

23. The Universal Data Factory Vision

When you strip away the MAL-specific configuration, this pipeline is a universal synthetic training data factory. The same architecture is used by Meta (Self-Instruct), Microsoft (Orca, Phi-4, AgentInstruct), and Scale AI at enterprise scale.

The Generalized Pipeline — Domain-Agnostic View
flowchart LR seedIntent["Seed Intent\n(any example)"] --> templateExpand["Template Expansion\n(any task variants)"] templateExpand --> llmSynth["LLM Synthesis\n(any generator)"] llmSynth --> qualGate["Quality Gate\n(any validator)"] qualGate --> registry["Training Registry\n(any dataset)"]

Only five configuration points tie this pipeline to MAL today. All five are configuration changes, not architectural rewrites:

ComponentCurrently MAL-SpecificGeneralization Path
Seed schemascenario_id, type, domain, input.intent → MAL outputGeneric intent → output pair with configurable output schema
Jinja2 templatesMAL syntax (entity, policy, workflow)Domain template library: /templates/{domain}/{task_type}.j2
Domain packstelecom.json with MAL vocabularyPlugin system: any domain drops into /domain_packs/{name}.json
Quality gateValidates MAL syntax, schema, compile_checkPluggable validators: JSON Schema / regex / LLM-as-judge
Registry recordoutput.mal fieldGeneric output.content with MIME type (code / SQL / text / JSON)

25. Generalization Architecture Roadmap

Three phases to evolve from MAL factory to universal factory. Each phase is self-contained — Phase 2.5 does not block Phase 3.

Phase 2.5 — Multi-Domain (3–6 months)
  • Add banking domain pack (fraud detection, KYC, loan processing scenarios)
  • Add healthcare domain pack (clinical workflow, HL7 FHIR scenarios)
  • Domain selector dropdown in frontend UI (telecom / banking / healthcare)
  • Target: 336 → 1,000+ training records across 3 domains from the same pipeline
  • No architectural changes — only new JSON domain packs and seed files
Phase 3 — Universal Factory (6–12 months)
  • Config-driven domain system — YAML domain packs, pluggable validators
  • LLM-as-judge quality gate (replaces MAL-specific compile_check for new domains)
  • CLI: alm-datagen generate --domain sql --seeds 100 --target 5000
  • Web UI: drag-drop seed file → pipeline runs → download JSONL dataset
  • First external use case: SQL generation training data for a code LLM
Phase 4 — Active Learning Loop (12–18 months)
  • Evaluation harness: benchmark model quality before and after each generation round
  • Uncertainty sampling: identify model failure modes → generate targeted gap-filling data
  • Automatic seed expansion: LLM reads domain documentation → generates new seeds automatically
  • Full closed loop: model gap → targeted data generation → fine-tune → eval → repeat

This pipeline independently arrived at the same design choices as leading research groups. The table below maps landmark papers to specific parts of the current implementation — providing academic grounding for the approach.

TechniquePaper & YearHow This Pipeline Maps

Evol-InstructXu et al. 2023 (WizardLM) “Synthesize All Seeds” at increasing complexity levels = WizardLM-style evolutionary instruction generation
OrcaMicrosoft 2023 System prompt grounding via domain packs = Orca’s explanation and rationale grounding for richer training signals
AgentInstructMicrosoft 2024 6 task types (Create / Modify / Repair / Explain / Policy / Disambiguate) = exact taxonomy match to AgentInstruct’s agent capability taxonomy
Phi-4Microsoft 2024 Quality Gate before registry = Phi-4’s multi-stage quality filtering before any data enters training
LIMAZhou et al. 2023 56 curated seeds as the anchor = LIMA’s proof that 1,000 carefully curated examples outperform 52,000 random ones; quality beats quantity
Key insightThis pipeline’s design was not reverse-engineered from these papers — it arrived at the same conclusions independently through practical implementation. That convergence validates the architectural choices and makes the research grounding authentic rather than retrofitted.
