16. Risk Register

High
LLM stage still not fully defined — Model choice, prompt strategy, and acceptance logic are all open. Keep this stage explicitly marked as proposed until each is resolved.
High
Invalid MAL risk — More volume creates more invalid outputs unless syntax and validation gates are automated before scaling generation.
Medium
Low-diversity dataset risk — Template expansion can become repetitive if payload generation is too narrow. Diversity criteria must be tracked, not assumed.
Medium
Review bottleneck — Human approval will slow the pipeline if scoring, batching, and rejection reasons are not standardized before scaling review.
Medium
Domain drift risk — UI or unrelated constructs can leak into the dataset if generation is not kept inside the MAL-only grammar scope.
Medium
Graph compile failure — Syntactically valid text can still fail downstream compile expectations. Graph checks must remain part of the acceptance funnel.
Planned
Registry discipline — Accepted records must move through explicit states (draft, validated, reviewed, accepted, archived) rather than accumulating in a flat file.

17. Recommendation

Go
Build the Jinja2 and Python expansion layer first. The most concrete, lowest-risk, and easiest-to-govern Phase 2 starting point. It gives the team a controlled synthetic-data baseline before introducing model-driven variation.
Keep
Reuse existing ALM validation and governance architecture. The repo already models validation, graph reasoning, policy checks, and versioning. Those ideas should drive the synthetic-data pipeline design, not be duplicated separately.
Next
Introduce LLM expansion only after the baseline is measurable. Once template expansion produces accepted data with clear metrics, the team can add model-driven variation with much stronger confidence and better comparison data.

18. Roadmap

Now
Phase 2 Foundation
Define Gold Seeds
Build a curated seed set covering all MAL task types and construct families across all four capability layers.
Design Jinja2 Templates
Convert seeds into structured templates with safe substitution zones.
Create Payload Generator
Use Python to generate approved payload combinations and emit deterministic variants.
Define Registry Contract
Set a simple but strict schema for accepted synthetic records and metadata.
Next
Controlled Expansion
Add Validation Pipeline
Apply syntax, schema, policy, graph, and duplicate checks automatically.
Introduce Review Workflow
Add scoring, acceptance states, reviewer notes, and promotion logic.
Measure Coverage
Track accepted count, diversity, rejection reasons, and construct coverage using Section 6 grid.
Pilot LLM Expansion
Run a controlled experiment with prompt versioning, domain packs, and strict filtering.
Later
Mature Data Factory
Registry as Source of Truth
Promote accepted data into a maintained dataset registry with full versioning.
Model Strategy Layer
Compare providers, prompting styles, and strategies by quality and cost using Section 12 methods.
Active Error Learning
Use rejection feedback loop (Section 10.4) to improve templates, payload rules, and prompts.
Efficient Fine-Tuning
Start LoRA-style adaptation experiments on the accepted MAL dataset.
