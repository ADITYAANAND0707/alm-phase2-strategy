Part I
Context & Direction
Strategic overview, scope definition, baseline gaps, and immediate priorities

1. Executive Summary

  • The ALM platform already provides runtime MAL generation, validation, graph compilation, and retrieval — but it has no offline synthetic-data pipeline for training or evaluation yet.
  • Phase 2 should stay MAL-specific and aligned to the current grammar: application, entity, workflow, policy, agent, integration, event, and evolution. UI constructs are out of scope for the first data wave.
  • The safest, most concrete first step is a Jinja2 + Python expansion layer that produces valid MAL variants by swapping approved values into known-good templates.
  • The LLM variant expansion stage should be introduced only after template expansion, validation gates, graph-compile checks, and registry discipline are all in place.
  • This document is a living technical report. New stages, model decisions, review metrics, and acceptance criteria should be added here as Phase 2 matures.

10. Execution Playbook

Each stage below should be treated as a distinct build target with its own acceptance criteria and completion signal before the next stage begins.

10.0 Layered Capability Design

The synthetic data strategy is built around four capability layers that define what the model must learn at each level of training data. Seeds and templates should consciously target each layer, not just the Create task type.

L1

Syntax & Structure

Block order, keywords, grammar patterns, and simple correction.

Data: intent → MAL block; malformed → corrected; partial → completed.

Goal: Parser-aligned MAL fluency.

L2

Task Behavior

Create, modify, explain, and repair behaviors across construct types.

Data: user intent → full MAL; change request → evolution; MAL context → explanation.

Goal: Aligns model with ALM operating modes.

L3

Domain & Policy

Telecom domain rules, business approvals, and policy constraints.

Data: intent + domain pack → domain-valid MAL; intent + constraints → policy-compliant MAL.

Goal: Safe, domain-correct output.

L4

Robustness & Recovery

Clarification handling, ambiguity, and recovery after failures.

Data: rejected output + validator feedback → corrected MAL; ambiguous request → confirmation-first flow.

Goal: Fewer production errors and repeat failures.

Planning direction Seed authoring and Jinja2 templates should be distributed across all four layers — not concentrated only on L1 Create examples. Covering L2–L4 is what makes the dataset production-grade.

10.1 Gold Seed Examples

The seed layer is the foundation of the full pipeline. Seeds should represent canonical input-output pairs for MAL creation or change tasks. The goal is correctness and coverage, not volume.

  • Target 40 to 60 clean seed examples as the starting range.
  • Cover distinct MAL construct families: entity, workflow, policy, integration, agent, event, evolution.
  • Keep the seed set versioned and manually curated.
  • Treat seeds as reference truth, not just examples.

10.2 Jinja2 Template Expansion

The first implementation target. Each seed example should be converted into a structured template with allowed substitution zones. Payload generation is handled in Python so every variant stays inside a safe, approved design space.

  • Use placeholders for names, enums, attributes, steps, integrations, roles, and policy options.
  • Keep business logic stable while varying legal values.
  • Generate JSON payloads separately from template files for auditability.
  • Export generated pairs in a registry-friendly structure.

10.3 LLM Variant Expansion

What We Know vs What We Don't Know Yet

What We Know

Stable Foundation

  • Template expansion works and produces syntactically valid MAL.
  • The validation pipeline (parser, schema, policy, graph) is already defined.
  • The provenance model (seed → template → registry) is clear.
  • The registry lifecycle states are agreed: draft → validated → reviewed → accepted → archived.
  • LLM expansion must come after, not before, the template baseline.
Build on this — do not skip it
What We Don't Know Yet

Open Research Questions

  • Which model gives the best balance of MAL structure-following and cost.
  • Whether direct instruction, few-shot, chain-of-thought, or repair prompting performs best.
  • How much diversity is useful before outputs become noisy or drift from grammar.
  • What score or rule set qualifies a candidate for human review.
  • What metadata must be stored now to support future fine-tuning use cases.
These must be answered before scaling LLM generation

This stage should be introduced only after template expansion and validation are stable. Its role is not to replace templates, but to extend diversity beyond deterministic slot filling. The correct goal is candidate generation under constraints, not free-form creativity.

What Can Be Done Now

Immediate LLM Tasks

  • Generate paraphrased user intents from approved seed examples.
  • Produce harder modify, repair, and ambiguity-handling instructions from clean seeds.
  • Create controlled MAL variants using fixed output format and strict construct scope.
  • Generate explanation-oriented pairs: MAL to explanation, validator error to repair suggestion.
Use the MAL Agent as a candidate generator only — not as final acceptance source
Prerequisites

What Must Exist First

  • Approved gold seed library with category coverage.
  • Stable MAL scope and construct rules.
  • Automated parser, validator, and graph-compile checks.
  • Prompt versioning, model version tracking, and dataset provenance fields.
  • Near-duplicate detection to prevent registry flooding.
Do not scale LLM generation before validation and provenance are operational

Domain Pack — What It Is and Why It Matters

Concept

What a Domain Pack Is

A domain pack is a versioned JSON file that defines the approved vocabulary for a specific domain (e.g., telecom). It is injected into the LLM prompt as a constraint, preventing the model from inventing field names, enum values, or roles that don't exist in the domain.

  • Prevents grammar drift and hallucinated field names.
  • Makes outputs directly compatible with policy and validator checks.
  • Should be maintained separately and versioned alongside templates.
Example
{ "domain": "telecom", "entity_names": [ "Subscriber", "SIMCard", "ServicePlan", "Invoice" ], "attribute_types": [ "UUID", "STRING", "INTEGER", "BOOLEAN", "ENUM", "DATE" ], "approved_enums": { "kyc_status": ["Pending","Verified","Rejected"], "subscription_type": ["Prepaid","Postpaid"] }, "approved_roles": [ "subscriber","agent","supervisor" ], "workflow_trigger_prefixes": [ "new_","update_","cancel_","verify_" ] }

Constrained MAL Prompt Template

What a production-ready constrained LLM prompt looks like for MAL candidate generation. This template format should be versioned and stored alongside the seed library.

Prompt Template — Constrained MAL Generation
SYSTEM:
  You are an expert MAL generator for the ALM platform.
  - Only produce MAL inside the allowed construct scope below.
  - Do NOT generate UI constructs (screens, components, nav).
  - Follow MAL grammar strictly: application, entity, workflow,
    policy, agent, integration, event, evolution.
  - Apply the domain pack for all field names and enum values.
  - Output format: raw MAL text only — no explanations or markdown.

CONSTRAINT SCOPE:
  allowed_constructs: [entity, workflow, policy]
  domain: {{ domain_name }}
  task_type: {{ task_type }}
  domain_pack_version: {{ domain_pack_version }}

REFERENCE SEED (approved — use as structural guide):
{{ seed_mal_text }}

USER INTENT:
{{ user_intent }}

INSTRUCTION:
  Generate a valid MAL program that fulfills the user intent.
  All entity attributes must come from the domain pack value set.
  The output must pass parser validation and graph compile checks.
  Do not add constructs outside the allowed_constructs list.

Recommended LLM Workflow

  1. Start from an approved seed or approved Jinja2-generated record.
  2. Build a constrained prompt with MAL-only scope, allowed constructs, domain pack, and output contract.
  3. Generate one or more candidate variants with the model.
  4. Run parser validation, domain and policy checks, and graph compile checks.
  5. Run novelty and duplicate filtering against the accepted registry.
  6. Send only surviving candidates to human review.
  7. Promote accepted records into the registry with source, model, and prompt metadata.

10.4 Validation and Filtering

Every candidate — whether produced by Jinja2 or an LLM — must pass layered quality gates before entering the accepted dataset.

Gate 1

Syntax and Schema

  • MAL syntax validity
  • Required fields present
  • Template payload completeness
Gate 2

Policy and Graph

  • Policy compliance checks
  • Graph-aware validation
  • Structural consistency checks
Gate 3

Quality and Duplication

  • Near-duplicate filtering
  • Diversity scoring
  • Low-value or noisy output rejection

Every rejection is pipeline intelligence. Reason codes must be tracked so template rules, prompt constraints, and seed quality can all be improved from real failure data — not just accepted record counts.

10.5 Human Review and Approval

The human review layer is what turns a generation pipeline into a professional dataset pipeline. Reviewers should not only accept or reject — they should tag why records were corrected or dropped.

Acceptance Criteria Checklist

CriterionPass ConditionReviewer Action if Fail
MAL syntax validityParser returns zero errorsReject — log syntax error category and source
Required fieldsAll mandatory construct fields are populatedReject — note which fields are missing
Policy complianceDomain policy check passes without warningsReject — flag specific policy violation
Graph compileGraph compiler accepts the MAL without errorsReject — log compile failure and construct type
NoveltyCosine similarity to existing registry records < 0.92Reject — mark as near-duplicate, reference closest match
CorrectnessOutput correctly fulfills the stated user intentCorrect and re-submit, or reject with correction note
DiversityIntent and output differ meaningfully from five nearest registry recordsAccept with diversity note, or reject if too similar
UsefulnessOutput teaches a distinct behavior the model should knowFlag as low-value — defer or reject
Domain accuracyAll field names, enums, and roles match the approved domain packCorrect using domain pack, or reject if drift is significant
  • Score on: correctness, diversity, usefulness, and policy fit.
  • Capture reviewer notes and correction reasons.
  • Promote only accepted data into the official registry.
  • Keep rejected samples for error analysis, not for training.

11. Model Adaptation Strategy

The strategy report recommends efficient fine-tuning on the accepted MAL dataset as the first serious model-adaptation step — not full-parameter retraining.

DimensionFull-parameter fine-tuningEfficient fine-tuning
What it updatesAll model weightsSmall trainable subset — adapters or LoRA-style methods
StrengthHighest adaptation capacityFaster, cheaper, easier to iterate
WeaknessExpensive and riskier with evolving dataLower adaptation ceiling than full retraining
Best fit for current Phase 2Not firstRecommended first step
Planning directionEfficient fine-tuning should only begin once the accepted dataset has measurable quality. The accepted registry is the prerequisite — not the experiment target.

16. Risk Register

High
LLM stage still not fully defined — Model choice, prompt strategy, and acceptance logic are all open. Keep this stage explicitly marked as proposed until each is resolved.
High
Invalid MAL risk — More volume creates more invalid outputs unless syntax and validation gates are automated before scaling generation.
Medium
Low-diversity dataset risk — Template expansion can become repetitive if payload generation is too narrow. Diversity criteria must be tracked, not assumed.
Medium
Review bottleneck — Human approval will slow the pipeline if scoring, batching, and rejection reasons are not standardized before scaling review.
Medium
Domain drift risk — UI or unrelated constructs can leak into the dataset if generation is not kept inside the MAL-only grammar scope.
Medium
Graph compile failure — Syntactically valid text can still fail downstream compile expectations. Graph checks must remain part of the acceptance funnel.
Planned
Registry discipline — Accepted records must move through explicit states (draft, validated, reviewed, accepted, archived) rather than accumulating in a flat file.

17. Recommendation

Go
Build the Jinja2 and Python expansion layer first. The most concrete, lowest-risk, and easiest-to-govern Phase 2 starting point. It gives the team a controlled synthetic-data baseline before introducing model-driven variation.
Keep
Reuse existing ALM validation and governance architecture. The repo already models validation, graph reasoning, policy checks, and versioning. Those ideas should drive the synthetic-data pipeline design, not be duplicated separately.
Next
Introduce LLM expansion only after the baseline is measurable. Once template expansion produces accepted data with clear metrics, the team can add model-driven variation with much stronger confidence and better comparison data.
